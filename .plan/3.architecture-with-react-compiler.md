## 3. Архитектура с учётом React Compiler

- [ ] **Принципы работы с React Compiler**
  - [ ] ✅ Писать чистый, предсказуемый код — Compiler сам мемоизирует там, где нужно
  - [ ] ✅ Использовать `useCallback`/`useMemo` **только** для:
    - Ссылок, которые должны сохранять идентичность (например, для `React.memo` детей)
    - Зависимостей в `useEffect`, где важна стабильность ссылки
  - [ ] ❌ Избегать преждевременной оптимизации — доверять Compiler'у
  - [ ] ❌ Не передавать в зависимости хуков не-примитивы без необходимости

- [ ] **Выделение зон ответственности**
  - [ ] `EntityTable` — "умный" контейнер: управляет состоянием `data`, `filters`, `sortState`
  - [ ] `EntityFormModal` — "глупый" презентационный компонент: `initialValues?`, `onSubmit`, `onCancel`
  - [ ] `SearchBar` — отдельный компонент с debounced-вводом
  - [ ] `TableActions` — кнопки с иконками, не зависят от внешнего состояния

- [ ] **Управление состоянием (без лишних библиотек)**
  - [ ] `useState` для: `data: EntityItem[]`, `searchQuery: string`, `modalState`
  - [ ] Кастомные хуки для инкапсуляции логики:
    - [ ] `useEntityData()` — CRUD-операции с иммутабельными обновлениями
    - [ ] `useTypedSort()` — типобезопасная сортировка по разным типам
    - [ ] `useDebouncedSearch()` — debounce 300ms через `setTimeout` + cleanup

- [ ] **Подготовка к асинхронности (на перспективу)**
  - [ ] Заложить интерфейс `EntityService` с методами `create`, `update`, `delete`
  - [ ] Использовать `useActionState` (React 19) для мутаций с оптимистичным UI
